# matenaiyo リーチ拡大戦略

作成日: 2026-07-08 / 状態: 立案(未着手)

## 前提認識(正直な現実)

- 「日程調整」というヘッドキーワードは調整さん・TimeRex・Spir 等が支配しており、
  新規個人開発サイトが検索1ページ目を短期で取るのは非現実的。
- 現実的な流入経路は優先度順に:
  1. **プロダクト内のバイラルループ**(幹事1人→参加者N人が製品を体験する構造)
  2. **AI検索(ChatGPT/Claude/Perplexity/Google AI Overview)での言及・推薦**
  3. **ロングテール SEO**(「日程調整 ログイン不要」「調整さん 代わり」等)
  4. 開発者コミュニティ経由(OSS・技術記事)
- Lighthouse スコアは「悪いと減点」だが「良くても加点は小さい」。基盤整備として最初に測って直し、以降は維持。

## 最初に決めるべきこと(後戻り困難)

**独自ドメイン。** 現在 `matenaiyo.vercel.app`。被リンク・AI の学習/引用・SNS シェアはすべてドメインに蓄積するため、
拡散施策を始める前に決めないと資産が引っ越せない。`matenaiyo.com` / `matenaiyo.jp` / `matenaiyo.app` 等。
vercel.app サブドメインはスパムサイトも多く、ドメイン単位の信頼評価で不利。
移行時は 301 リダイレクトを Vercel 側で設定。

## 柱1: 技術 SEO 基盤(1〜2日、全施策の前提)

| 項目 | 内容 |
|---|---|
| robots.ts | `/api/` のみ Disallow。`/e/` は Disallow **しない**(robots.txt でブロックすると noindex メタタグが読まれず、外部リンク経由で URL がインデックスされうる — Google 仕様) |
| noindex | `/e/[slug]` 系全ページに `robots: { index: false, follow: false }`。参加者名が載る UGC ページのインデックスはプライバシー事故になる。**SEO 以前に必須** |
| sitemap.ts | トップ + 静的コンテンツページのみ列挙(イベントページは含めない) |
| JSON-LD | トップに `WebApplication`(name, description, offers: 無料, operatingSystem: Web)+ `FAQPage` |
| canonical | metadataBase 済み。独自ドメイン移行時に更新 |
| Search Console | 登録・sitemap 送信。効果計測の土台 |
| README | テンプレのまま。GitHub もランディングの一つ(特に開発者流入)。スクショ・機能・技術スタック・デモ URL を記載 |

## 柱2: LP / コンテンツ

### トップページの LP 化(既存フォームは維持)

現トップは「即作成フォーム」でコンバージョンには最適。壊さずファーストビュー下に追加:

- 使い方 3 ステップ(スクショ or 図)
- 機能紹介(カレンダー UI・リアルタイム集計・確定→カレンダー連携・ダークモード)
- **FAQ セクション**(JSON-LD `FAQPage` と対応)— 「ログインは必要?」「無料?」「回答の修正は?」「データはいつ消える?」
- 調整さんとの機能比較表(名指しの攻撃はせず事実ベース)

`page.tsx` 全体が `"use client"` なので、Server Component の親ページ + フォームだけ client island に分離すると保守しやすい(SEO 上は Next のプリレンダリングで HTML は出るため必須ではない)。

### ユースケース別 LP(ロングテール狙い、効果が出るまで3〜6ヶ月)

`/for/bonenkai`(忘年会)、`/for/nomikai`(飲み会)、`/for/zemi`(ゼミ・研究室)、`/for/hogoshakai`(保護者会)等。
各ページ = ユースケース固有の悩み + 使い方 + そのままフォームへ。
狙うクエリ: 「忘年会 日程調整 ツール」「飲み会 日程 決め方 アプリ」など。
最初は 2〜3 ページで反応を見る。量産は Search Console のデータを見てから。

## 柱3: AI SEO(LLMO / GEO)

AI 検索は「新しくてもコンテンツが明瞭なら推薦されうる」ため、後発には従来 SEO より公平な戦場。

1. **llms.txt** — サイト概要・機能・使い方をプレーンテキストで `/llms.txt` に。効果は確立していないがコスト極小
2. **FAQ・構造化された事実記述** — AI は「ログイン不要」「無料」「広告なし」等の明確な事実を引用しやすい。曖昧なコピーより仕様の明記
3. **第三者ページでの言及を増やす** — AI は複数ソースの一致を重視。Zenn/Qiita 記事、比較記事、GitHub README が引用元になる
4. **OSS であることを武器に** — GitHub 公開済みは差別化。'調整さんの OSS 代替' というポジションは AI が比較回答で拾いやすい。GitHub Topics(`scheduling`, `nextjs`, `supabase`)設定
5. 「調整さん 代替」「調整さん ログイン不要 代わり」等の比較クエリは AI 検索で頻出。比較コンテンツを自サイトに持つ

## 柱4: バイラルループ強化(最重要・最速)

このプロダクトの構造的な強み: **幹事1人が使うと参加者 N 人が必ずページを見る**。調整さん自身がこの構造で普及した。

1. **回答完了画面に「自分も matenaiyo で日程調整を作る」CTA** — 参加者が幹事に転換する瞬間を設計する。現状ヘッダーロゴしか導線がない
2. イベントページのフッターに小さく「Powered by matenaiyo — ログイン不要の日程調整」
3. OGP は実装済み(イベント名・回答数入り)— シェアされるたびにブランド露出。維持
4. 確定通知・.ics にもサービス名が残るように

計測: GA で「参加者ページ → /new 遷移」をイベント計測し、ループの転換率を追う。

## 柱5: パフォーマンス / Lighthouse

1. **まず計測**(PageSpeed Insights / `pnpm dlx lighthouse`)。推測で最適化しない
2. 予想される論点:
   - 日本語 Web フォント 2 種(Zen Maru + Zen Kaku)は容量が大きい。`display: swap` 済みだが、ウェイト削減や見出しのみ Web フォント化を検討
   - トップが全体 client component — LCP への影響を計測で確認
3. Core Web Vitals は Search Console + Vercel Speed Insights で継続監視
4. 目標: Performance 90+ / SEO・A11y・Best Practices 100(現構成なら大工事は不要のはず)

## 柱6: SNS 以外の一次拡散チャネル

- **Zenn / Qiita に開発記事**(「Next.js 16 + Supabase Realtime で調整さんを作り直した」系)— 個人開発ツールの初期流入で最も再現性が高い。記事→GitHub→サービスの導線
- はてなブックマーク圏に届けば一時的スパイク + 被リンク獲得
- 個人開発系サービスへの登録(運営者ギルド系リスト等)
- 比較記事・まとめ記事を書いているブロガーは新ツールを常に探している — 掲載は自然発生を待つより README とスクショを整備して見つけやすくする
