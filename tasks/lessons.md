# Lessons

## 設定スクショは「後戻り不可の項目」を最初のレビューで指摘する

**状況**: Supabase の New project 画面スクショで Region が「Asia-Pacific」になっていたのを最初は素通りし、プロビジョニング後(=リージョン変更不可になった後)にムンバイ割当を指摘してしまった。ユーザーから「最初のスクショの時点で言ってほしかった」と指摘。

**教訓**: 設定・作成系の画面(スクショ含む)をレビューするときは、まず **後戻りが効かない/高コストな項目** を最優先でチェックし、その場で指摘する。特に:
- **リージョン / ロケーション**(作成後変更不可なことが多い。「Asia-Pacific」等のグループ表記は Tokyo 以外=Mumbai 等に解決され得る → 日本向けなら `ap-northeast-1` を明示選択と言う)
- 可視性 public/private、課金プラン、プロジェクト名/URL、暗号化・認証方式など一度決めると変えにくいもの

**適用**: 「作成」「New」「Create」ボタンが見えるフォームのスクショが来たら、押す前提で全フィールドを走査し、ユーザーのユースケース(このプロジェクトは日本向け日程調整)に照らして不適切/曖昧な既定値を先に洗い出してから、次の操作を案内する。

## 同一 worktree で複数エージェントを並行稼働させない

**状況**: 実装委譲した background subagent が「作業せず終了した」ように見えたが、実際は内部で子エージェントを起動して待機していた。作業なしと判断して再指示した結果、同一 worktree 上で2エージェントが並行稼働し、後から終了した子エージェントが親セッションの tasks/*.md 更新を「身に覚えのない変更」として git checkout で巻き戻した(コード差分は無事)。

**教訓**:
- subagent の完了通知が来ても、報告文が曖昧なら再指示の前に「本当に停止しているか」(子エージェントの有無・output の tool_uses/duration)を確認する
- 委譲プロンプトの MUST NOT DO に「他のエージェントを起動しない」を明記する
- subagent 完了後に自分がファイルを編集する場合、別エージェントがまだ生きている可能性があるなら編集を保留するか、完了後に git status で巻き戻りを検証する

**適用**: 委譲 → 検証 → ドキュメント更新の流れでは、最後に必ず `git status` / 該当箇所 grep で最終状態を確認してから完了報告する。
